# 趋势优先仓位管理设计文档

## 概述

本设计用于解决当前自动交易引擎“过早卖出强趋势股票，随后又高位买回”的问题。根因不是单次买回本身，而是持仓决策过于粗糙：当前系统对已有持仓只有 `sell` 和 `hold` 两种动作，且固定止盈会直接全卖，导致本应继续持有、减仓锁利或顺势加仓的场景，被误判为清仓离场。

本次设计将交易引擎从“买卖触发器”升级为“趋势优先的仓位管理器”。系统的首要目标不再是尽快止盈，而是优先吃住主升浪：趋势未坏时尽量保留仓位，短线过热时优先减仓，趋势增强时允许加仓，只有趋势破坏时才清仓。

## 问题定义

### 当前行为

当前综合交易流程位于 `web_app.py` 的 `_comprehensive_trade()`：

1. 先执行硬性止损/止盈
2. 再调用 `ai_comprehensive_decision()` 生成持仓与开仓建议
3. 对持仓只支持 `sell / hold`
4. 对候选股只支持新开仓 `buy`

当前实现存在以下问题：

- **持仓动作过少**：AI 无法表达“加仓”或“减仓”，只能“卖”或“继续拿着”。
- **固定止盈过硬**：到达止盈阈值后直接全卖，不适合“吃主升浪”的目标。
- **卖出语义过重**：很多本该 `hold` 或 `reduce` 的场景被迫落成 `sell`。
- **后续买回只是表象**：系统卖出后再次买回，说明前面的清仓判断本身就不够智能。

### 用户目标

本次设计以以下目标为准：

1. 优先吃住主升浪
2. 减少过早清仓和卖飞
3. 允许顺势加仓
4. 允许在短线过热时减仓而非直接清仓
5. 仅在趋势破坏时退出

## 设计原则

### 1. 盈利不是清仓理由，趋势破坏才是

系统不应因为“已经盈利”就直接退出。盈利只代表进入利润管理区，不代表上涨逻辑结束。只有当趋势结构破坏、风险显著放大、原持仓理由失效时，才应触发清仓。

### 2. 默认优先持有

在没有明确减仓、加仓、退出信号时，默认动作应为 `hold`。新的系统要学会“少做错误交易”，而不是“每轮都做动作”。

### 3. 只允许顺势加仓，不允许逆势摊平

加仓仅用于趋势强化、突破确认、仓位仍低且风险预算允许的场景。对浮亏且走弱的持仓，不允许通过加仓摊低成本。

### 4. 止损硬、止盈软

- 硬止损保留，用于账户底线保护。
- 固定止盈改造为“利润管理区”，到达后不直接清仓，只提升 `reduce` 的优先级。

## 新的仓位动作模型

### 持仓动作

对每只已有持仓，AI 只能输出以下四类动作之一：

- `add`：加仓
- `hold`：继续持有
- `reduce`：减仓
- `exit`：清仓

### 动作含义

#### add
用于趋势强化的股票。执行结果是在现有持仓上继续买入，提高该标的在账户中的权重。

#### hold
用于趋势健康但没有明确加减仓信号的股票。执行结果是不发生交易。

#### reduce
用于短线偏热、仓位偏重、冲高回落、放量滞涨但趋势尚未完全破坏的股票。执行结果是卖出一部分仓位并保留底仓。

#### exit
用于趋势明显破坏、止损触发、原持仓逻辑失效的股票。执行结果是清掉全部可卖仓位。

## AI 决策合同调整

### 当前合同问题

当前 `ai_comprehensive_decision()` 返回的 `position_actions` 只支持 `sell / hold`，表达力不足。

### 新合同

`position_actions` 升级为仓位管理格式：

```json
{
  "symbol": "000725",
  "action": "add/hold/reduce/exit",
  "target_pct": 35,
  "change_pct": 10,
  "reason": "趋势延续且量价配合改善，建议提高仓位",
  "confidence": 82
}
```

说明：

- `target_pct`：该标的目标仓位，占总权益百分比
- `change_pct`：相对当前仓位建议变化值，便于日志和前端展示
- `reason`：简洁说明动作依据
- `confidence`：辅助过滤极低质量建议，不直接取代硬风控

### 新开仓合同

`buy_picks` 保留，但语义改成“仅针对尚未持有的标的推荐开新仓”。对于已持有股票的增强判断，应统一通过 `position_actions` 的 `add` 表达，避免同一只股票同时出现在“持仓动作”和“新开仓推荐”里。

## 执行层设计

### 总体思路

执行层不再直接理解为“AI 说买一笔 / 卖一笔”，而是：

1. 计算当前总权益
2. 计算每只持仓当前仓位占比
3. 读取 AI 的目标仓位建议
4. 将当前仓位调整到目标仓位附近

### 当前仓位计算

当前仓位 = `该标的当前市值 / 账户总权益`

其中总权益仍复用现有 `calc_current_equity()` 逻辑。

### hold 执行

- 不发生实际交易
- 仅记录日志和决策原因

### add 执行

- 计算目标仓位对应的目标市值
- 计算当前市值与目标市值之间的差额
- 将差额换算为买入股数
- 按整手规则调用现有 `exec_buy()`

约束：

- 不超过 `max_position_pct`
- 不超过现金上限和每日买入限额
- 单次加仓金额不超过预设上限
- 对亏损走弱持仓不执行加仓

### reduce 执行

- 计算应降低的目标市值
- 换算为卖出股数
- 调用现有 `exec_sell()` 卖出部分仓位

约束：

- 只卖出可卖数量（保留 T+1 限制）
- 避免因差额太小产生碎片化交易
- 减仓后若剩余仓位过小，可按阈值决定不执行或直接转为清仓逻辑（第一版优先采用“不执行小额减仓”）

### exit 执行

- 卖出全部可卖数量
- 行为等价于当前 AI 卖出分支，但门槛更高

## 风控设计

### 保留硬止损

现有硬止损继续保留。若亏损达到止损阈值或出现明确定义的风险线破位，即使 AI 倾向持有，也应优先保护账户。

### 改造硬止盈

当前固定止盈线不再直接触发全卖。改为：

- 达到 `take_profit_pct` 后，标记为“进入利润管理区”
- 在 AI prompt 中明确：高盈利优先考虑 `reduce` 或 `hold`
- 只有趋势失效才允许 `exit`

这意味着：

- `take_profit_pct` 的含义从“止盈卖出线”改为“利润管理阈值”
- 高盈利只是提高 `reduce` 权重，而不是自动清仓

### 加仓防守线

第一版必须避免越加越满，因此需要新增或强化以下约束：

- 单票目标仓位不得超过 `max_position_pct`
- 单次加仓变动建议限制在总权益的 10% 左右
- 同一持仓在一次 tick 内最多执行一次仓位变更
- 对低信心或理由不足的 `add` 直接降级为 `hold`

## 第一版动作判定框架

### exit
出现以下任一类强信号：

- 硬止损触发
- 趋势结构明显破坏
- 原持仓理由失效

### reduce
不满足 `exit`，但满足以下任一类：

- 已有较大浮盈且短线偏热
- 价格显著偏离均线
- 冲高回落
- 放量滞涨
- 单票仓位过重

### add
不满足 `exit / reduce`，且同时满足：

- 趋势继续强化
- 当前仓位不高
- 风险预算和现金允许
- 非逆势补仓

### hold
不满足以上条件时默认 `hold`

## 非目标

本次设计不包含以下内容：

1. 不引入复杂的多模型打分系统
2. 不做高频日内反复微调
3. 不通过大量冷却期和买回限制补丁解决问题
4. 不做大规模架构拆分，优先在现有单文件结构中增量改造

## 测试策略

第一版需要覆盖以下核心行为：

1. 盈利达到利润管理区但趋势未坏时，不再自动清仓
2. AI 输出 `reduce` 时只卖部分仓位
3. AI 输出 `add` 时在现有持仓基础上加仓，并正确重算均价
4. 趋势破坏或止损触发时仍能清仓
5. 仓位调整遵守 `max_position_pct`、现金约束、整手规则和 T+1 约束
6. 不允许对亏损走弱持仓执行“摊平式加仓”

## 成功标准

以下结果可视为本设计达标：

1. 强趋势股票不再因为固定止盈被直接全卖
2. 系统能对持仓输出 `add / hold / reduce / exit`
3. 减仓与加仓成为一等动作，而不是临时补丁
4. 系统在主升浪场景下更倾向保留底仓
5. “卖飞后再高位买回”的频率显著下降

## 实施建议

建议按以下顺序推进：

1. 先完成 AI 输出合同与执行层改造
2. 再重构固定止盈的语义
3. 最后补充测试与日志展示

这样可以优先完成“从二元买卖到仓位管理”的核心升级，避免一开始就在附加优化上分散精力。
