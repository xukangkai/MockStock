# Agent 主导选股设计文档

## 概述

当前系统虽然已经使用 Agent 参与选股与综合决策，但程序侧在进入 Agent 前做了较强的候选收缩：全市场行情先按成交额筛到前 100，再只对前 10 做增强，候选猎手提示词又只消费前 8，并额外叠加固定价格区间与“只推荐买得起”的强规则。这使得系统看起来像是“Agent 在选股”，但实际上大量选股偏好已经被程序提前决定，最终容易反复落到少数低价高流动性个股上。

本次设计的目标不是彻底移除程序约束，而是把系统调整为“中度放权”模式：程序负责收集市场数据、整理候选摘要、维持账户与执行层风控；Agent 负责在更大的市场候选范围内自主判断什么标的更适合买入。这样既保留系统的安全边界和执行可靠性，也让“买什么”尽量成为 Agent 的真实判断，而不是预筛逻辑的副产物。

## 问题定义

### 当前问题

当前候选路径存在三层会显著限制 Agent 自由度的收缩：

1. 候选池虽然取了成交额前 100，但只对前 10 做增强并传给 Agent。
2. 候选猎手提示词只展示前 8 个候选条目。
3. 候选猎手提示词强制要求股票价格处于固定区间，并提前强调“只推荐买得起的”，使模型天然偏向低价且可一手成交的股票。

这些限制叠加后，即使全市场中存在更多适合短线交易的个股，Agent 也往往只能在一个很窄的候选面内做判断。结果是系统“看起来由 Agent 选股”，但买入结果仍被程序结构强烈引导。

### 用户目标

用户希望系统改成：

1. 候选范围扩大到前 100 只活跃股。
2. 程序不再通过固定价格区间或极窄候选列表替 Agent 预先决定答案。
3. Agent 在更大的候选面上自己判断更适合买什么。
4. 程序仍保留账户、仓位、下单合法性与执行时段等硬风控。

## 方案比较

### 方案 A：单层放权，直接把前 100 候选摘要交给候选猎手

让候选猎手直接消费前 100 活跃股的轻量摘要，综合判断最值得买入的标的，再交给总指挥整合风控意见输出最终计划。

优点：

- 最贴近“把买什么交给 Agent 判断”的目标。
- 工程改动相对最小，可在现有架构上直接演进。
- 能显著降低程序预筛造成的单票偏置。

缺点：

- Prompt 变长，模型处理成本会上升。
- 候选摘要设计不好时，模型可能在大量候选中判断不稳定。

### 方案 B：两阶段 Agent 粗筛 + 精筛

先让一个节点从前 100 候选中选出 10-15 只 shortlist，再由候选猎手或总指挥在 shortlist 上做精筛。

优点：

- 模型更容易处理，稳定性更高。
- 可逐步细化分析过程。

缺点：

- 引入额外 AI 调用，复杂度与时延更高。
- 仍有一层中间筛选，不如单层放权直接。

### 方案 C：程序做轻量排序，Agent 在加权候选中做最终决策

程序先计算一份对小资金账户友好的轻量排序，再把排序结果和全部候选一起交给 Agent，由 Agent 自主裁决。

优点：

- 比当前系统自由得多，同时比纯放权更稳定。
- 对小资金账户适配更友好。

缺点：

- 程序偏好依然较重，不完全符合“尽量交给 Agent 判断”的方向。

### 方案选择

本次设计选择 **方案 A：单层放权，直接把前 100 候选摘要交给候选猎手**。

原因如下：

1. 用户明确希望“程序只负责数据收集和执行风控，Agent 从更大市场池里自己挑”。
2. 当前系统的主要问题不是执行层，而是候选输入过窄、提示词过强地预先决定了答案。
3. 方案 A 在满足用户目标的同时，对现有架构侵入最小，便于快速验证效果。

## 设计原则

### 1. 程序负责准备题面，不负责替 Agent 先做结论

程序仍可筛出“前 100 活跃股”作为可处理的市场切片，但不应继续通过前 10、前 8、固定价格区间等规则把答案提前缩成少数低价股。

### 2. 账户与执行风控必须保留在程序侧

Agent 可以自主判断候选优劣，但最终下单前仍必须经过程序侧的现金、仓位、一手成交、交易时段等校验。Agent 的自主性不能覆盖这些硬约束。

### 3. 候选摘要必须轻量、统一、可比较

前 100 候选不适合全部携带完整历史和财务详情。程序需要为每个候选构造结构化轻量摘要，让模型能横向比较，而不是只看到几个被过度包装的对象。

### 4. 日志必须解释“Agent 为什么选这只”

一旦放权，用户更需要知道买入标的是如何从 100 只候选中胜出的，以及执行层是否又在后面做了拦截。因此日志与调试信息必须同步增强。

## 总体设计

### 新的选股数据流

改造后的数据流如下：

1. 获取全市场行情。
2. 继续按成交额筛出前 100 只活跃股。
3. 为前 100 只构造轻量候选摘要。
4. 候选猎手基于前 100 摘要自主筛选和推荐。
5. 总指挥结合市场分析、持仓审视与风控意见输出最终计划。
6. 执行层对最终买入建议执行资金、仓位与交易时段校验。

与当前实现相比，最核心变化是：Agent 不再只消费前 8-10 个被严重收缩的候选，而是基于前 100 只活跃股形成自主判断。

## 模块调整

### 1. 候选构建逻辑

当前 `_comprehensive_trade()` 会先得到前 100 活跃股，但只对前 10 个候选追加更多指标并传入 Agent。改造后应拆成两个层次：

- **轻量摘要层**：对前 100 只全部生成统一的结构化摘要。
- **增强信息层**：保留已有技术/财务增强逻辑，但不再作为限制 Agent 输入范围的门槛。

轻量摘要建议至少包含：

- `symbol`
- `name`
- `price`
- `amount`
- `change_pct`
- `is_etf`
- `ma5`（若可得）
- `pe` / `pb`（若可得）
- `net_profit_yoy` / `revenue_yoy`（若可得）
- 其他已有但计算成本低、适合横向比较的字段

这些摘要应统一放入 `candidates_ctx`，供候选猎手直接消费。

### 2. 候选猎手提示词

当前候选猎手提示词会：

- 只展示前 8 条候选。
- 固定要求股票价格在 5-50 元、ETF 在 0.5-5 元。
- 强调“只推荐买得起的”，提前把资金约束注入选股逻辑。

改造后应调整为：

- 明确说明当前输入是“成交额前 100 的活跃股候选摘要”。
- 不再限制固定价格区间。
- 不再把“价格×100 ≤ 可用现金”作为候选猎手的前置硬门槛，而是改成“请结合账户规模、流动性和风险收益比，优先选择更适合当前账户规模的标的”。
- 允许候选猎手输出 0-3 只真正看好的标的，而不是为了满足规则硬找低价票。

候选猎手的职责应变成“从更大候选面中做研究判断”，而不是“执行程序预先定义好的买得起/低价过滤器”。

### 3. 总指挥节点

总指挥节点总体职责不变，仍负责：

- 综合市场分析结果
- 综合持仓动作建议
- 综合候选猎手推荐
- 结合风控建议输出最终计划

但在改造后，总指挥将收到更具“Agent 自主筛选”意味的候选结果，因此提示词里不需要再额外鼓励它只从低价票里挑选。总指挥继续聚焦组合层面的取舍：该不该买、买几只、留多少现金、风险偏向如何。

### 4. 执行层

执行层继续保留现有硬校验，包括：

- 最大持仓数限制
- 可用现金限制
- 一手成交限制
- 单次买入金额上下限
- 非交易时段仅记录、不真实下单
- 行情可得性校验

这部分逻辑不应因本次放权而削弱。它是“Agent 决策自由”与“系统执行安全”之间的最后边界。

## 日志与可解释性

为避免放权后出现“为什么又选了某只票却解释不清”的问题，日志应增强为：

1. 记录本轮输入候选总数。
2. 记录候选池前若干只预览。
3. 记录候选猎手最终产出数量。
4. 当候选猎手未推荐任何标的时，记录其拒绝理由。
5. 当总指挥或执行层否决买入时，明确标注是研究层未推荐、综合层收敛为不开仓，还是执行层因资金/仓位/行情规则拦截。

这样既能让用户理解 Agent 的选股逻辑，也能帮助后续继续迭代候选摘要与提示词。

## 测试与验证

本次改造后，至少需要验证以下行为：

1. Agent 输入候选不再被限制为前 8-10 只，而是来自前 100 活跃股。
2. 候选猎手提示词中不再包含固定价格区间约束。
3. 在小资金账户下，Agent 仍能自主推荐高于当前账户可一手成交价位的个股，但最终执行层会安全拦截不可执行计划。
4. `/api/picks`、`/api/agent/status`、`/api/agent/cycles` 和实时日志仍能正确展示本轮推荐与原因。
5. 当候选猎手没有看到合适标的时，系统依然可以安全输出空仓决策。

## 非目标

本次设计不包含以下内容：

- 不引入第二轮 AI 粗筛/精筛流程。
- 不重写执行层资金与交易规则。
- 不把候选范围扩展到全市场完整输入。
- 不新增长期记忆驱动的推荐去重与行业分散机制。

这些可以作为后续增强项，但不属于本次“中度放权，让 Agent 在前 100 活跃股中主导选股”的实现范围。
