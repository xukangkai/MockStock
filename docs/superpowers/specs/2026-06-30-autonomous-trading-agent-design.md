# 自主决策交易 Agent 设计文档

## 概述

当前系统已经具备自动交易闭环：引擎可定时拉取市场数据、生成候选股票、调用 AI 做综合判断，并执行买卖与仓位管理。它的主要短板不是“没有 AI”，而是决策能力仍停留在单次调用：缺少多步推理、缺少工具调用、缺少历史记忆、缺少角色分工，也缺少对决策过程的结构化审计。

本次设计的目标，是将现有系统从“单次 AI 决策函数”升级为“可自主决策的交易 Agent 系统”。升级后的系统既保留当前成熟的执行与风控主干，也新增多节点协作、结构化记忆、结果反馈与前端可视化，使其更像一个持续学习、可解释、可追踪的投研与执行 Agent，而不是一个把 prompt 写得更长的交易脚本。

本次设计采用“研究与执行兼顾，但第一阶段优先强化执行”的方向：系统既要具备更强的市场研究与推理能力，也要优先服务于自动决策和自动交易闭环，不追求一开始做成完全自由协商式多 Agent，而是优先落地可控、可调试、可渐进演进的主控编排架构。

## 问题定义

### 当前系统能力

当前项目的核心能力主要集中在以下部分：

1. `AutonomousTradingEngine` 负责交易时段心跳运行
2. `ai_comprehensive_decision()` 负责对持仓和候选股生成综合建议
3. `exec_buy()` / `exec_sell()` 负责真实的模拟成交、手续费、T+1 和 lot 匹配
4. FastAPI + 前端页面负责账户展示、决策日志、推荐列表与收益曲线

这些能力已经覆盖“自动扫描—自动判断—自动执行—自动展示”的完整链路。

### 当前主要不足

尽管已有自动交易闭环，当前系统仍存在以下结构性问题：

- **单次决策过于集中**：几乎所有策略推理都塞在一次 AI 调用内，表达力和可控性不足。
- **缺少多步思考流程**：系统不会先判断市场环境、再看持仓、再审查候选、再做风险复核，而是一次性给结论。
- **缺少工具自主调用能力**：AI 只能消费预先整理的数据，不能根据情境主动搜索新闻、查询额外信息或检索历史经验。
- **缺少记忆与学习闭环**：系统不会系统化地记住最近几天哪些操作有效、哪些决策导致回撤。
- **缺少角色分工**：市场判断、个股研究、风险审查、执行决策全部由一个 AI 同时承担，不利于约束与扩展。
- **可解释性不足**：当前前端能看到结果，但看不到各步骤如何得出结论，也无法清晰区分分析意见与风控否决。

### 用户目标

本次升级以以下目标为准：

1. 引入更强的多步推理链路
2. 让系统具备短期记忆和日终复盘能力
3. 让 Agent 能主动调用工具，而不仅仅被动接受固定输入
4. 采用多角色或多节点协作，而不是继续依赖单个 AI 大 prompt
5. 在提升研究深度的同时，优先保证交易执行闭环可用、稳定、可回退

## 方案比较

### 方案 A：单 Agent 增强版

在现有 `ai_comprehensive_decision()` 上升级成单 Agent 循环，让一个主模型按“思考—查工具—观察—决策”的形式工作。

优点：

- 改动最小
- 最容易嵌入现有系统
- 第一版最快上线

缺点：

- 很难体现真正的角色分工
- 风控和研究仍容易耦合在一起
- 后续再扩展成多 Agent 时仍需要重构

### 方案 B：完全多 Agent 协作版

把系统拆成市场分析、个股研究、风险控制、交易执行、复盘学习等多个自治 Agent，以自由协商或状态图方式完成联合决策。

优点：

- 最贴近“自主决策 Agent”的理想形态
- 分工清晰，后续扩展空间大
- 更利于分别优化研究、风险和执行策略

缺点：

- 改动较大
- AI 调用成本高
- 状态流更复杂，调试难度明显增加

### 方案 C：主控 Agent + 专家节点的混合演进版

使用 LangGraph 组织状态流，由一个主控 Agent 负责编排，多位专家节点提供结构化意见，最终由统一的决策综合层输出可执行交易计划。

优点：

- 兼顾多节点分工与工程可控性
- 可以保留当前执行主干与数据库模型
- 适合从现有单体项目逐步演进，不需要一开始大拆文件
- 更利于把记忆、日志、前端展示串成闭环

缺点：

- 不如完全多 Agent 自由协商那样激进
- 某些更复杂的自治行为要留待后续阶段再扩展

### 方案选择

本次设计选择 **方案 C：主控 Agent + 专家节点的混合演进版**。

选择原因如下：

1. 当前项目已经有完整的自动交易闭环，最值得升级的是“决策大脑”，而不是执行引擎。
2. 用户明确希望系统具备推理链、记忆、工具使用和多角色协作，但同时也希望优先保证交易执行质量。
3. LangGraph 适合把单次 AI 调用改造成可观察、可回退、可插拔的状态图，而无需立即推翻现有结构。
4. 这种架构便于后续渐进增强：第一阶段先做主控编排，第二阶段再增加更复杂的专家协作、事件感知与长期记忆能力。

## 设计原则

### 1. 保留成熟执行主干，优先替换决策大脑

`AutonomousTradingEngine`、持仓模型、成交执行、账户快照、手续费计算、T+1 约束等能力已经成熟。升级时应优先替换决策组织方式，而不是重写执行层。

### 2. 决策过程必须结构化，而不是继续堆大 prompt

升级目标不是写一个更长的 `ai_comprehensive_decision()` prompt，而是把市场判断、持仓诊断、候选研究、风控复核、计划生成拆成明确节点，并定义输入输出合同。

### 3. 研究增强必须服务执行闭环

本次方向是“研究与执行兼顾，但先偏执行”。因此新增能力要优先提升交易计划质量、仓位控制、风控解释和结果复盘，而不是先追求开放式问答或泛化投研助手。

### 4. 记忆只能形成偏置，不能推翻硬风控

Agent 的复盘经验可以影响风险偏好、仓位建议和候选排序，但不能覆盖最大仓位、止损底线、现金校验、T+1 规则和执行合法性检查。

### 5. 自主性必须建立在可审计之上

系统可以自主决定重点看什么、调哪些工具、如何平衡进攻与防守，但每轮过程都必须记录为可阅读的结构化日志，以便用户理解、调试和追责。

## 总体架构

### 分层设计

升级后的系统建议拆成四层：

#### 1. Execution Layer（执行层）

保留现有交易执行能力，包括：

- `AutonomousTradingEngine`
- `exec_buy()`
- `exec_sell()`
- 账户、持仓、lot、快照与手续费逻辑

该层职责从“兼做策略思考”收缩为“接收交易计划并执行”。

#### 2. Agent Orchestration Layer（Agent 编排层）

新增统一入口，例如 `run_trading_agent_cycle(context)`。该层负责：

1. 收集和标准化上下文
2. 调用记忆和工具层
3. 编排专家节点执行顺序
4. 汇总节点输出，生成最终交易计划
5. 写入日志和记忆

该层建议使用 LangGraph 构建状态流。

#### 3. Specialist Layer（专家层）

第一阶段至少包含以下专家节点：

- 市场分析节点
- 持仓诊断节点
- 候选股研究节点
- 风险审查节点
- 复盘与记忆节点

这些节点不直接下单，只输出结构化意见，最终由综合层统一决策。

#### 4. Tool & Memory Layer（工具与记忆层）

负责为 Agent 提供“可调用能力”和“可回忆经验”，包括行情、财务、新闻、历史决策、复盘摘要等信息来源。

## 核心模块设计

### 1. AgentContextBuilder

职责：

- 汇总账户、持仓、候选股、市场快照、最近交易、最近决策、历史记忆摘要
- 生成标准化的 `AgentCycleContext`

该模块的目标是把原本分散在各处的输入，统一整理成一份“决策包”。

### 2. TradingAgentGraph

职责：

- 定义 LangGraph 主状态流
- 管理一轮 Agent 生命周期
- 控制节点顺序、依赖与分支条件

建议节点顺序：

1. `build_context`
2. `recall_memory`
3. `analyze_market`
4. `review_positions`
5. `research_candidates`
6. `risk_review`
7. `synthesize_decision`
8. `validate_plan`
9. `execute_trades`
10. `persist_results`

### 3. MarketAnalysisNode

职责：

- 判断当前市场属于进攻、平衡、防守还是恐慌环境
- 提供风险偏好和方向提示

典型输出包括：

- market regime
- 情绪评分
- 风险偏向
- 主攻方向与规避方向
- 全局警告

### 4. PositionReviewNode

职责：

- 对每只现有持仓生成结构化诊断
- 给出 `add / hold / reduce / exit` 的建议动作
- 解释趋势、风险和仓位理由

### 5. CandidateResearchNode

职责：

- 对候选股票做排序和筛选
- 输出推荐买入、继续观察和明确放弃的标的
- 结合技术面、财务、估值、新闻与市场环境适配度给出理由

### 6. RiskReviewNode

职责：

- 独立审查各类建议，防止研究节点与主控节点过度乐观
- 检查仓位上限、板块集中度、市场环境适配性与回撤期纪律
- 产出 `pass / warn / block / adjust` 结果

### 7. DecisionSynthesizer

职责：

- 汇总市场、持仓、候选和风控结果
- 形成统一的 `FinalTradingPlan`
- 保证下游执行层可以直接消费，而不是再解释自然语言

### 8. MemoryManager

职责：

- 读取短期和长期记忆
- 记录本轮决策摘要与结果反馈
- 生成日终复盘摘要

### 9. AgentAuditLogger

职责：

- 记录每轮与每个节点的输入摘要、输出摘要、耗时、工具调用、模型信息和执行结果
- 为前端的决策链展示提供数据来源

## Agent 状态流设计

### 单轮流程

每个交易周期按以下顺序执行：

1. `Build Context`
2. `Recall Memory`
3. `Analyze Market`
4. `Review Positions`
5. `Research Candidates`
6. `Risk Review`
7. `Synthesize Decision`
8. `Validate Plan`
9. `Execute Trades`
10. `Persist Logs & Memory`

### 各节点职责

#### Build Context

收集并标准化本轮所有输入，不做判断，只做信息整理。

#### Recall Memory

根据市场环境、板块标签、setup 类型和最近结果，检索相关短期经验和近期复盘摘要，为后续节点注入策略提醒。

#### Analyze Market

判断市场应偏进攻、平衡还是防守，为整轮决策设定总基调。

#### Review Positions

逐只审查持仓，输出对既有仓位的动作建议与原因。

#### Research Candidates

研究候选标的，对新开仓机会进行排序、筛选和风险提示。

#### Risk Review

独立检查过度集中、逆势加仓、回撤期激进开仓等风险，并对建议进行通过、警告、阻止或降级调整。

#### Synthesize Decision

整合所有节点，产出一份统一的 `FinalTradingPlan`。

#### Validate Plan

在交给执行层之前做程序化校验，确保不违反 T+1、现金、整手、仓位上限和状态一致性要求。

#### Execute Trades

调用现有 `exec_buy()` / `exec_sell()` 执行交易，尽量复用现有成交逻辑。

#### Persist Logs & Memory

记录本轮过程、最终计划、实际执行结果，并为后续复盘和记忆检索提供数据。

### 条件分支与降级策略

第一阶段采用“线性主流程 + 少量条件分支”。重点条件分支如下：

1. **市场极弱**：可跳过候选研究，优先进入持仓防守模式。
2. **当前空仓**：可弱化持仓诊断，强化候选研究。
3. **风控拦截**：如风险过高，本轮禁止新增仓位，只允许 `hold / reduce / exit`。
4. **工具或节点失败**：记录失败并降级使用已有结构化数据，避免整轮崩溃。
5. **风控节点失败**：默认进入保守模式，禁止新增开仓。
6. **最终计划非法**：回退为安全计划或本轮不交易。
7. **Agent 全部不可用**：退回到简化模式，只保留硬止损与安全逻辑。

## 工具设计

### 第一阶段工具范围

本次设计建议第一阶段仅接入以下工具：

1. **行情工具**：复用现有 Sina 行情抓取能力
2. **历史数据工具**：复用现有 K 线与技术指标计算能力
3. **财务与估值工具**：复用现有 akshare 财务与估值数据能力
4. **新闻/公告搜索工具**：为 Agent 提供事件驱动感知能力
5. **历史决策与交易检索工具**：从本地数据库中检索相似情境下的近期经验

### 工具使用原则

- 不是每轮都调用全部工具，而是根据情境按需触发
- 个股异动、持仓回撤放大或候选股出现异常时，再触发新闻或事件搜索
- 工具输出要以结构化摘要形式写入状态，而不是把冗长原始文本直接继续传下去

## 记忆系统设计

### 1. 运行期记忆

运行期记忆仅在当前 cycle 内存在，用于共享中间结论、已调用工具、已排除标的和风险限制。该层由图状态本身承载，不单独落库。

### 2. 短期记忆

短期记忆覆盖最近 1 到 5 个交易日或最近 10 到 30 个决策轮次，建议记录：

- 最近几天的市场 regime 与情绪变化
- 每轮最终计划与关键理由
- 风控拦截内容
- 某类 setup 的近期成功或失败表现
- 买入、减仓、退出后的短期结果反馈

这层记忆的目标是避免 Agent 重复犯错，并让其具有近期市场适应性。

### 3. 长期记忆

长期记忆按日或按周生成结构化复盘，总结被验证过的经验，例如：

- 弱情绪震荡日追高效果较差
- 主升期核心龙头低吸优于跟风票追涨
- 回撤期应主动降低单票目标仓位

第一阶段不引入复杂向量数据库，优先使用标签化与结构化检索。

### 记忆约束

1. 记忆不能覆盖硬风控
2. 单日结果不能直接沉淀为长期规律
3. 记忆只作为偏置，不作为绝对规则
4. 复盘与执行分层，Agent 不得自行改写系统底层纪律

## 学习闭环设计

### 1. 反馈采集

需要为关键动作建立后验反馈，例如：

- 买入后的 1 日 / 3 日 / 5 日表现
- 减仓是否降低后续回撤
- 清仓是否有效规避了风险
- 放弃买入是否错失明显机会

### 2. 复盘提炼

每天收盘后，系统将原始反馈提炼为结构化复盘对象，包括：

- 当日市场环境
- 有效模式
- 无效模式
- 纪律偏差
- 对下一交易日的风险偏向建议

### 3. 下一轮注入

在 `Recall Memory` 节点中，将与当前环境最相关的经验注入上下文，使 Agent 能带着近期经验进入下一轮决策。

## 新增数据表设计

### agent_cycle_logs

记录每轮 Agent 运行总览，包括：

- `cycle_id`
- 时间
- 是否成功
- 总耗时
- 最终摘要
- 风险等级
- 是否触发交易

### agent_node_logs

记录每个节点的执行摘要，包括：

- `cycle_id`
- `node_name`
- `input_summary`
- `output_summary`
- `model_name`
- `tool_calls`
- `duration_ms`

### agent_memories

记录结构化记忆，包括：

- `memory_type`
- 日期
- 标签
- `content_json`
- 相关度或权重字段（可选）

### agent_feedback

记录动作的后验结果，包括：

- 动作类型
- 目标标的
- 1 日 / 3 日 / 5 日表现
- 是否改善回撤或盈亏比
- 对应的 cycle_id 和决策摘要

## 前端展示设计

当前前端已具备账户、持仓、交易、推荐和日志展示能力。升级为 Agent 后，前端应新增以下视图：

### 1. Agent 决策链视图

按 cycle 展示完整流程卡片或时间线，包括：

- 时间
- 市场判断
- 持仓诊断摘要
- 候选研究摘要
- 风控意见
- 最终计划
- 实际执行结果

### 2. Agent 记忆面板

展示最近对当前决策产生影响的关键记忆，例如：

- 最近高位追涨表现不佳
- 某板块持续性不足
- 减仓策略在弱市中效果较好

### 3. 复盘与学习面板

独立页面或 tab 展示：

- 当日正确和错误决策案例
- 风控拦截次数
- 经验总结
- 对下一日的偏置建议

### 展示边界

前端展示结构化摘要、关键理由、风险提示和工具结果摘要，不直接展示原始 prompt 或冗长思维链。

## 第一阶段范围

第一阶段聚焦以下能力：

1. 从单次 AI 调用升级为多步 Agent 状态流
2. 引入市场分析、持仓诊断、候选研究、风控审查四个核心节点
3. 接入基础工具调用能力
4. 实现短期记忆与日终复盘摘要
5. 前端展示 Agent 决策链与记忆摘要

## 非目标

第一阶段不包含以下内容：

1. 不做完全自由协商式多 Agent 辩论系统
2. 不让 Agent 自动修改底层风控参数或执行纪律
3. 不引入大规模向量数据库或复杂 RAG 基建
4. 不做高频日内多轮自适应交易引擎
5. 不对现有项目做大规模文件级重构，优先在现有架构中增量演进

## 测试策略

第一阶段至少需要覆盖以下行为：

1. Agent graph 能在正常场景下生成合法 `FinalTradingPlan`
2. 当市场极弱时，系统能进入防守模式并禁止新增激进开仓
3. 当风控节点阻止交易时，最终计划会正确降级
4. 当某个工具调用失败时，系统仍能优雅降级而不是整轮崩溃
5. 短期记忆和日终复盘能被下一轮检索并影响决策偏向
6. 前端可以正确展示 cycle、节点摘要和记忆摘要
7. 执行层仍遵守现金、仓位、T+1、整手和现有交易约束

## 成功标准

以下结果可视为本次设计达标：

1. 系统不再只依赖单次 AI 综合调用，而是具备清晰的多步决策流
2. Agent 能根据情境自主调用工具并输出结构化结论
3. 系统能记住近期经验，并在下一轮决策中体现偏置
4. 风控审查从单一 prompt 内隐逻辑升级为独立可观察节点
5. 前端能展示 Agent 的分析链路、记忆摘要和最终计划
6. 在保留现有执行主干的前提下，整体自主决策能力显著增强

## 实施建议

建议按以下顺序推进：

1. 先抽取上下文构建与最终计划数据结构
2. 再搭建 LangGraph 主流程和四个核心专家节点
3. 然后接入工具层、短期记忆和节点日志
4. 最后补充前端展示与日终复盘能力

这样的顺序可以优先建立最关键的决策骨架，再逐步把记忆、展示和学习闭环补齐，避免一开始把所有子系统同时铺开。
